Mysql 的锁与MVCC

一、mysql的锁

行锁（Record lock）

记录锁就是为某行记录加锁。例如

锁住的是id为2这一行记录。id 列必须为「唯一索引」列或「主键」列，否则上述语句加的锁就会变成「临键锁（next-key）」。

间隙锁（Gap Lock）

锁定索引记录间隙，锁定与该key相邻两个键之间的空白区域，确保索引记录的间隙不变。间隙锁是针对事务隔离级别为可重复读「RR」或以上级的。

间隙锁产生条件

  • 使用普通索引锁定
  • 使用多列唯一索引
  • 使用唯一索引锁定多行记录

    对于使用唯一索引来搜索并给某一行记录加锁的语句，不会产生间隙锁。（这不包括搜索条件仅包括多列唯一索引的一些列的情况；在这种情况下，会产生间隙锁。）例如，如果id列具有唯一索引，则下面的语句仅对具有id值100的行使用记录锁，并不会产生间隙锁

2）唯一键索引间隙锁场景2

此例中，id=8正好落在(1,10]中，其他事务中，id=8,9,5都会被阻塞。即产生了(1,8]和(8,10]的间隙锁和id=8的行锁

结论：当主键使用的是一条不存在的记录，那么会加行锁和间隙锁。

3）普通索引产生间隙锁场景

同样使用上述数据表，测试使用age字段，该字段是普通非唯一索引。该字段隐含的间隙锁如下

  • (-无穷,1]

  • (10,20]

此例中，id=10，间隙锁(1, 10]和(10,20]会产生作用，所以id=8和id=12的插入记录都会被阻塞，直到事务1提交后才能继续执行

结论:在普通非唯一索引上，只要加锁都会产生间隙锁

临键锁(next-key lock)

next-key lock就是行锁和间隙锁的组合，他锁定的范围既包括当前行，也包括间隙区间。

二、MVCC

全称：multy version concurrent control

前言

  • MySQL 中 InnoDB 引擎支持 MVCC
  • 应对高并发事务, MVCC 比单纯的加行锁更有效, 开销更小
  • MVCC 在读已提交（Read Committed）和可重复读（Repeatable Read）隔离级别下起作用
  • MVCC 既可以基于乐观锁又可以基于悲观锁来实现

MVCC 原理

  • 在Innodb中，每一行记录都有两个隐藏列：

  • DATA_TRX_ID「最近一次对本行记录做修改的事务标识符，即最后一次修改本行记录的事务ID，6个字节」；

  • DATA_ROLL_PTR「表示指向该行回滚段（rollback segment）的指针，7个字节」

  • DB_ROW_ID：行标识（隐藏单调自增 ID），大小为 6 字节，如果表没有主键，InnoDB 会自动生成一个隐藏主键，因此会出现这个列

  • Undo Log的组织

  • 一致性读实现方案
    在 RU 隔离级别下，直接读取版本的最新记录就 OK，对于 SERIALIZABLE 隔离级别，则是通过加锁互斥来访问数据，因此不需要 MVCC 的帮助。

    • RR隔离级别的read view
      在RR隔离级别下，「每个事务开始时」，只执行第一个select语句，后面所有的select都复用这个Readview，即第一次生成快照，后面的查询皆从快照出，其他update、delete、insert语句和一致性读snapshot建立没有关系，将当前系统中所有活跃事务拷贝到一个列表生成readview。
      RR级别下MVCC 的readvew生成参考图

    • RC隔离级别下的read view
      在 RC 隔离级别下，「每个 SELECT 语句开始时」，都会重新将当前系统中的「所有的活跃事务」拷贝到一个列表生成 ReadView。

readview读取过程

  • 关于流程图中t_max，也就是readview的上届，有人说t_max是readview中活跃事务的最大id，有人说是未被使用的事务id的最小值（因为事务id一直自增，未被使用的最小id一定是以系统分配的、提交后+活跃事务中最大事务id+1），个人更倾向后者。
